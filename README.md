## How I tend to work

- I want to understand the problem before I open the editor.
- I prefer boring, typed, testable code over clever code.
- I pick tools that fit the team, not tools that look good on a CV.
- I write things down — PRs, ADRs, READMEs — so the next person isn't stuck.
